HTTP Splitting/Smuggling Explained

HTTP Response Splitting

HTTP Response Splitting occurs when an application reflects user input in HTTP response headers without proper validation. Attackers can inject CRLF (%0D%0A) sequences to split the response and inject additional headers or content.

  • CRLF Injection: Using %0D%0A to inject new headers
  • Header Manipulation: Injecting Set-Cookie, Location, or Content-Type headers
  • Cache Poisoning: Storing malicious content in web caches
  • Session Fixation: Forcing users to use attacker-controlled session IDs

HTTP Request Smuggling

HTTP Request Smuggling exploits discrepancies in how different servers (frontend/backend) parse HTTP requests, particularly when Content-Length and Transfer-Encoding headers conflict.

  • CL.TE Smuggling: Frontend uses Content-Length, backend uses Transfer-Encoding
  • TE.CL Smuggling: Frontend uses Transfer-Encoding, backend uses Content-Length
  • Bypass Security Controls: Smuggling requests to bypass authentication or WAF
  • Session Hijacking: Capturing other users' requests/responses

Remediation Recommendations

To prevent HTTP Splitting/Smuggling vulnerabilities, implement the following measures:

Input Validation and Sanitization

  1. Filter CRLF Characters: Remove or encode %0D%0A sequences from user input
  2. Validate Headers: Ensure user input doesn't contain header-like patterns
  3. Whitelist Approach: Only allow known good input patterns
  4. Content-Type Validation: Validate and restrict content types

Secure HTTP Processing

  1. Consistent Parsing: Ensure all servers parse HTTP requests consistently
  2. Header Normalization: Normalize conflicting headers before processing
  3. Disable Unnecessary Features: Disable chunked encoding if not required
  4. Use Modern HTTP Versions: Prefer HTTP/2 which eliminates many smuggling issues

Web Server Configuration

  1. Web Application Firewall: Implement WAF rules to detect splitting/smuggling attempts
  2. Load Balancer Configuration: Ensure consistent header processing across infrastructure
  3. Request Monitoring: Monitor for unusual HTTP patterns
  4. Security Headers: Implement proper security headers to prevent injection
